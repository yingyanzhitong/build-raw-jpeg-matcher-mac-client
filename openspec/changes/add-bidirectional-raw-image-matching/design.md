## Context

当前应用采用 React/TypeScript 前端与 Tauri/Rust 本地后端。配对链路从类型、状态、命令到界面均按单向“JPEG 输入 → RAW 候选 → 导出 RAW”建模：前端保存 jpegInputs、rawSourceDirectory 和 selectedRawFormats，Rust 暴露 collect_jpeg_inputs、match_raw_files、export_raw_files，并在 MatchResult 中使用 jpeg 与 selectedRaw 等方向专属字段。

现有文件选择器和 Rust 已支持 JPG/JPEG，但不支持 PNG；拖放监听作用于整个 WebView，所有拖入路径都会进入 JPEG 收集逻辑。首次引导包含“先选 JPG、再选 RAW 目录”的步骤，但常驻界面没有步骤编号或连接关系，更换 RAW 目录也不会使旧结果失效。

本变更需要同时改动 UI 状态、Tauri 命令协议、文件扫描、匹配、预览、冲突复核、导出和测试。整个工作流继续在本地运行，不新增网络服务或外部数据存储。

## Goals / Non-Goals

**Goals:**

- 支持“图片 → RAW”和“RAW → 图片”两个方向，并默认保持现有图片找 RAW 行为。
- 在左侧栏标题下方提供清晰、可访问的方向切换器；上方始终是输入文件，下方始终是查找目录。
- 支持大小写不敏感的 JPG、JPEG、PNG 和现有十二种 RAW 格式。
- 使用一套方向中立的数据模型、匹配状态、冲突复核和导出协议，避免复制两套工作流。
- 为两个方向隔离当前会话状态，并保证任何匹配条件变化都会使旧结果失效。
- 在输入与查找目录均有效时自动开始当前方向的查找，免除额外的手动确认步骤。
- 保持精确、可预测的文件名匹配和不覆盖目标文件的导出安全策略。

**Non-Goals:**

- 不进行 RAW 解码、格式转换或生成新的 JPG/PNG。
- 不引入 EXIF、拍摄时间、编辑后缀剥离或物理文件的模糊匹配。
- 不支持一个冲突结果同时确认并导出多个候选；同名多个图片或 RAW 仍按一对一冲突复核。
- 不将两个方向的任务状态持久化到应用重启后的会话。
- 不新增并行执行两个方向任务或取消正在运行的 Rust 文件系统任务。

## Decisions

### 使用方向中立的领域模型

新增 MatchDirection，取值为 ImageToRaw 与 RawToImage。前后端共享的核心概念改为：

    MatchFile
      path / fileName / baseName / extension / size / modifiedTime / manual

    MatchResult
      input / status / candidates / selectedCandidate

    DirectionWorkspaceState
      inputs / manualRefs / searchDirectory / selectedRawFormats
      results / logs / exportDirectory / exportReport / busy

Tauri 匹配和导出协议显式携带 direction，Rust 根据方向选择受信任的输入与候选扩展名集合。不得允许前端传入任意扩展名绕过后端白名单。

备选方案是保留现有 JPG→RAW 命令，再新增一套 RAW→图片命令。该方案会重复扫描、状态、冲突和导出逻辑，后续修复容易只覆盖一个方向，因此不采用。

### 将方向切换器实现为侧栏顶部的独立控制行

左侧输入区保持纵向阅读顺序：

    ┌────────────────────────────┐
    │ 当前方向：图片 → RAW  [切换]│
    └────────────────────────────┘
    ┌────────────────────────────┐
    │ ① 当前方向的输入文件       │
    └────────────────────────────┘
    ┌────────────────────────────┐
    │ ② 当前方向的查找目录       │
    └────────────────────────────┘

方向切换器放在侧栏标题下方的紧凑控制行，仅显示当前方向和蓝色的左右交换图标。交换图标使用悬浮提示和无障碍标签说明目标方向，默认不显示“切换为 …”文案；紧凑图标按钮保持可识别性，同时不与两张任务卡片的主操作竞争。两张卡片之间只保留稳定留白。

切换方向时重新配置两张卡片内容，而不是让固定卡片之间的箭头改为向上。ImageToRaw 显示“图片输入 → RAW 查找目录”，RawToImage 显示“RAW 输入 → 图片查找目录”；RAW 格式控件始终跟随 RAW 卡片。

步骤编号用于表达真实顺序，但第二步仅进行视觉降级，不强制禁用目录选择。这样既强化引导，又不限制熟悉工具的用户预先选择目录。

备选方案包括把分段控件放入标题栏，或在卡片之间只放交换图标。标题栏已有统计、日志和更新操作，窄窗口空间不足；单图标无法表达当前方向和切换结果，因此不采用。

### 使用短暂状态机表达上下角色交换

前端增加仅用于视图的 `DirectionTransition`：`from`、`to`、`phase` 和 `sequence`。一次切换固定分为两个阶段：旧卡片离场 120ms；中点更新 `activeDirection`；新卡片入场 200ms，总时长约 320ms。上方卡片向下离场、下方卡片向上离场，新角色以相反方向进入；顶部切换器同步更新当前与目标方向。

过渡存在时，`MatcherBusy` 保持原语义，不将视觉动画伪装成文件系统任务。另行锁定方向切换、文件/目录选择、拖放、RAW 格式修改、查找、导出、清空与冲突确认，避免操作在中点被路由到另一方向。两个 `DirectionWorkspaceState` 不移动、不清空，只在中点切换当前引用。

`prefers-reduced-motion` 下不创建延时状态机，直接切换方向、恢复按钮焦点，并通过 `aria-live` 宣告新方向。常规动画结束后同样将焦点还给交换按钮，确保键盘用户不会丢失操作位置。

### 每个方向保存独立的会话状态

RawMatcherWorkspace 保存 activeDirection 和按方向索引的两个 DirectionWorkspaceState。切换方向只切换当前状态引用，不复制、交换或重新解释路径。

收集输入、匹配或导出期间禁用方向连接器。清空操作只清当前方向；标题统计、结果、日志、导出报告和冲突弹窗均读取当前方向状态。

任何输入、查找目录或 RAW 格式变化统一调用 invalidateMatchState，清除当前方向的 results、冲突确认和 exportReport。切换方向本身不清除状态，用户返回时恢复该方向任务。

备选方案是在每次切换时清空并二次确认。该方案实现简单，但方向切换是核心能力，持续丢失任务会造成明显摩擦，因此不采用。

### 两步条件就绪后自动执行匹配

前端用纯函数统一判断当前方向是否已有至少一个输入、一个查找目录、非空 RAW 格式集合且没有运行中任务。文件、目录、拖放或文本清单成功加入新输入时，或用户选择了不同的查找目录时，若判断为就绪，立即记录“自动开始查找”日志并调用既有 `match_counterpart_files` 命令。

RAW 格式修改仍只使结果失效，不自动重复扫描，避免用户连续调整格式时产生多次目录遍历。工作区保留低优先级的“重新查找”操作，供用户在不改变输入或目录时显式复查。自动调用与手动调用共享同一个忙碌锁和匹配函数，因此不会并发或跨方向路由任务。

### 由 Rust 提供统一格式能力

Rust 维护唯一的 IMAGE_EXTENSIONS 与 RAW_EXTENSIONS 定义，并通过 matcher_capabilities 命令向前端返回受支持格式。文件选择器、格式筛选和文案从该能力结果派生，避免前端八种、后端十二种 RAW 格式继续不一致。

图片格式为 jpg、jpeg、png；RAW 格式为 cr2、cr3、nef、arw、raf、orf、rw2、dng、rwl、pef、3fr、iiq。扩展名比较统一转为小写。

RAW 的现有最小文件大小校验应用于物理 RAW 输入和 RAW 候选，不应用于图片。图片和 RAW 目录扫描均跳过已知 macOS 元数据目录。两个方向都提供文本清单：RAW→图片的清单可填写 RAW 文件名、路径或末尾编号；带 RAW 扩展名时必须属于当前启用的格式，未带扩展名的编号引用允许继续参与后缀匹配。

当 RAW→图片方向取消某个 RAW 格式时，前端从当前输入集合中移除该格式文件、记录日志并使结果失效。重新启用格式不会自动恢复已移除文件，用户可重新选择文件或目录。

### 复用一个方向化匹配引擎

新增方向中立命令：

    collect_match_inputs(direction, paths, selectedRawFormats)
    match_counterpart_files(direction, inputs, manualRefs, searchRoot, selectedRawFormats)
    export_matched_files(direction, results, exportDir, searchRoot, selectedRawFormats)

匹配时，Rust 根据方向递归扫描查找目录并按 baseName 建立候选索引。物理文件主文件名保持当前大小写敏感的精确匹配；不移除 -Edit 等后缀。两个方向的手工文本引用均先精确匹配，没有结果时再执行不区分大小写的主文件名后缀匹配。

候选按规范化路径稳定排序。零个候选为 missing，一个候选为 matched 并自动选择，多个候选为 conflict 且不自动选择。RAW→图片同时发现同名 JPG、JPEG、PNG 或跨目录图片时同样进入 conflict，只允许确认一个候选。

备选方案是按照扩展名优先级自动选择图片候选。该行为可能静默丢弃用户需要的文件版本，且与现有 RAW 冲突规则不一致，因此不采用。

### 预览和冲突复核根据文件角色渲染

结果表使用“预览、对应文件、状态、输入文件、候选、操作”的方向中立列结构。FilePreview 根据扩展名选择语义，但 RAW 与 JPG/JPEG/PNG 都通过 Rust 调用 Quick Look 生成受控缓存缩略图，再从 Tauri asset scope 已允许的缓存目录加载。这样无需把用户任意文件路径暴露给 WebView，也不扩大资源协议权限。

冲突弹窗左侧始终为输入，右侧始终为候选。弹窗标题、说明和确认按钮从 directionConfig 派生。具有真实路径的输入与候选可用系统默认应用打开；任一方向的手工文本引用没有真实路径时只展示引用信息，不渲染打开或预览操作。

### 导出按唯一候选路径执行并逐项容错

export_matched_files 只处理 matched 与 confirmed 的 selectedCandidate，并先按规范化候选路径去重。多个输入指向同一候选时只处理一次。由于 results 来自 WebView，后端在导出前使用 searchRoot 和 selectedRawFormats 重新扫描当前可信候选集合，并复用匹配规则验证输入与候选关系；不得只依赖前端提交的 candidates 自证归属。

导出继续扁平化到单一目标目录并保留原文件名：

- 目标不存在：以原子 create-new 方式创建并复制，增加 copiedCount。
- 目标存在且内容相同：不复制，增加 alreadyPresentCount。
- 目标存在但内容不同：不覆盖，增加 collisionCount。
- 源不存在、不可读或复制失败：增加 sourceErrorCount，记录错误并继续其他候选。
- missing 与未确认 conflict 分别增加对应跳过数量。

后端同时拒绝查找目录外候选、已取消格式、失效文件和候选符号链接。导出目标若为普通文件以外的路径（包括有效或悬空符号链接）一律视为 collision；目标创建使用不覆盖的原子语义，避免检查与复制之间的竞态覆盖。

### 使用方向配置统一界面文案和行为

前端定义 directionConfig，集中提供输入名称、候选名称、卡片标题、目录标题、支持格式说明、重新查找/导出按钮、空状态、结果列、冲突复核、快捷键提示和日志前缀。工作区将“匹配结果”、结果数、帮助提示和操作按钮合并到单一工具栏，避免重复展示“匹配工作台”和当前方向；方向本身继续由侧栏切换器表达。

页面标题与副标题不再占用第二条横栏，实时统计以无边框的轻量浮层置于工作区右下角，不侵入原生 macOS 导航栏，从而保留整个标题栏的窗口拖动行为。日志抽屉在工作区左右两侧保留相同边距，不再紧贴侧栏分割线。

标题栏使用单个默认高度（24px）的自绘栏，避免系统标题栏与网页标题栏叠成两行。自绘栏保留红黄绿窗口控件；有更新时，仅在窗口控件右侧显示蓝底白字“更新”。更新按钮和三个窗口控件是独立交互区，不带拖动标记；其余标题栏区域使用 Tauri 的拖动区域并启用最小化、关闭、缩放和拖动所需权限，保证窗口可从标题、留白和未占用区域拖动。

首次引导升级存储版本，并根据当前方向生成步骤内容。常驻步骤状态不依赖首次引导是否已读。

## Risks / Trade-offs

- [两个方向保存完整状态会增加前端状态复杂度] → 使用明确的 DirectionWorkspaceState、纯函数更新和统一失效函数，避免散落的条件分支。
- [前后端协议一次性泛化可能造成字段不一致] → 先增加新命令和共享 TypeScript 类型，再切换 UI，最后删除旧命令；为序列化字段添加契约测试。
- [大型目录在两个方向都可能扫描较慢] → 继续在 Rust 中执行递归扫描，仅在两步条件由用户完成时自动运行；RAW 格式调整不自动扫描，并保持忙碌状态与日志反馈。
- [恢复旧方向状态时文件可能已移动或失效] → 在重新匹配、打开和导出时验证路径，逐项报告失效文件。
- [同名多种图片全部进入冲突会增加复核次数] → 保持一对一规则的可预测性；未来若有明确需求，再独立设计多选导出。
- [顶部切换器增加侧栏高度] → 使用约 48px 的独立控制行并保留侧栏滚动，不压缩文件投放区和格式控件的可点击尺寸。
- [上下交换动效可能在切换中误路由输入] → 使用独立 `DirectionTransition` 锁定所有会修改当前方向状态的入口，并只在 120ms 中点切换活动引用。
- [统一展示十二种 RAW 格式会增加控件高度] → 保持紧凑网格或可访问的展开区域，格式定义仍由后端统一提供。

## Migration Plan

1. 在 Rust 中增加方向中立模型、格式能力、输入收集、匹配和导出命令，同时保留旧命令供现有前端运行。
2. 添加双方向 Rust 测试和序列化契约测试，确认 JPG/JPEG/PNG、十二种 RAW、冲突和安全导出行为。
3. 将前端类型与工作区状态迁移为 MatchDirection 和 DirectionWorkspaceState，并接入新的 Tauri 命令。
4. 增加顶部方向切换器、动态卡片、常驻步骤状态、两步就绪后的自动查找、方向化结果与冲突复核。
5. 完成快捷键、引导、导航栏统计、合并工作台栏、日志抽屉与导出反馈的方向化，并进行 macOS 手动验证。
6. 前端完全切换后移除旧的方向专属命令与类型。

本变更不涉及持久化数据迁移。若需要回滚，可恢复旧前端调用与旧命令注册；在旧命令删除前始终保留可运行回退点。

## Open Questions

无阻塞问题。多候选批量导出、跨重启保存两个方向任务，以及 EXIF/模糊匹配均留待独立需求评估。
