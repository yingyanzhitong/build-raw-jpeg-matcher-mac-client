## Context

三个工作区已经把运行日志汇总到 `App`，并共用一个底部日志面板开关，但导出结果仍由各工作区自行展示。匹配工作区还有一个仅服务于“按钮不可用”原因的局部提示，不能覆盖异步导出结果。Rust 返回的三个汇总结构都包含可明确判断实际失败的计数字段。

## Goals / Non-Goals

**Goals:**

- 统一三个工作区的导出结果反馈语义和样式。
- 由发起任务的工作区根据自身汇总结构生成准确文案。
- 由 `App` 统一管理 Toast 生命周期，并在失败反馈出现时展开底部日志。
- 底部日志打开时始终把滚动视口对齐到最新记录。
- 通过纯函数集中定义成功、失败和取消的反馈判定，便于测试。
- 让三个工作区使用同位置、同语义的清空操作和右下角状态摘要。

**Non-Goals:**

- 不改变 Rust 导出接口、返回结构或文件处理规则。
- 不把普通扫描、预览、打开目录等错误纳入本次导出 Toast。
- 不移除匹配工作区现有的按钮不可用原因提示和导出完成详情弹窗。

## Decisions

1. 新增共享的 `ExportFeedback` 类型和三个汇总转换函数。每个函数只接收对应导出汇总并返回 `success | warning | error` 与中文文案；调用异常由工作区构造 `error` 反馈。相比在 `App` 识别三种不同汇总，这能保持业务字段含义靠近各功能，同时让判定可独立测试。
2. `App` 持有单个导出 Toast 状态并向三个工作区传入 `onExportFeedback`。收到新反馈时替换旧 Toast；收到 `error` 时同时调用 `setLogPanelOpen(true)`。相比三个工作区分别实现 Toast 和日志展开，这一方案能确保视觉、自动消失和失败行为一致。
3. 实际失败字段决定失败：匹配使用 `sourceErrorCount`，分离使用 `failedCount`，水印使用 `failedCount`。同名跳过、冲突和未找到继续体现在文案/报告中，但不自动判为失败；水印仅取消且无失败时使用 `warning`。
4. Toast 使用固定定位、`role=status`/`role=alert` 与 `aria-live`，成功和警告自动展示约 3.2 秒，失败展示约 5 秒。失败同时打开日志，因此 Toast 不承担完整错误详情。
5. `ScrollArea` 向调用方暴露可选的视口引用；`LogBottomSheet` 在打开状态或最新日志变化后，把视口 `scrollTop` 设置为 `scrollHeight`。相比倒序渲染，保留时间从旧到新的阅读顺序，同时实现终端 `tail` 的最新记录跟随体验。
6. 匹配工作区现有的 `RotateCcw` 清空入口和确认弹窗作为交互基准。分离清空来源、输出目录、扫描结果、报告和日志；水印清空照片目录、样片、预览缓存、导出进度、报告和日志，但保留水印素材、文字、字体与三画幅持久化参数。运行中禁止清空，确认弹窗必须明确不会删除磁盘文件。
7. `App` 继续作为跨工作区状态展示层。分离状态上报总文件、图片、RAW、跳过数量；水印状态上报图片、横图、竖图、方图数量。状态浮层复用匹配工作区现有表面与数字样式；水印始终避让缩略图工具条，分离和水印在出现导出页脚时进一步上移。

## Risks / Trade-offs

- [快速连续完成多个任务时单 Toast 会被后一次反馈替换] → 应用当前每个工作区同一时间只运行一个导出任务；采用递增 ID 保证计时器不会误关新 Toast。
- [部分失败仍可能同时显示原有“导出完成”报告] → 保留详情报告，但 Toast 明确使用失败样式并自动打开日志，避免把“调用完成”误解为“全部成功”。
- [切换工作区会关闭日志面板] → 失败发生时工作区仍是当前任务页，立即展开当前日志；沿用既有切换行为，不额外持久化面板状态。
- [持续跟随会让用户在任务运行时难以停留查看旧记录] → 本次按用户明确要求采用严格 `tail`；任务结束后不再产生新日志，用户仍可自由向上查看。
- [状态浮层可能遮挡工作区底部控件或结果页脚] → 由工作区状态上报结果页脚是否存在，按当前工作区使用固定避让高度，不修改内容布局。
