## Context

当前应用是 Tauri v2 + React + TypeScript + Rust 的本地 macOS 桌面工具，主界面集中在 `src/App.tsx`，后端命令集中在 `src-tauri/src/lib.rs`。现有 RAW/JPEG 匹配流程已经包含本地目录选择、递归扫描、日志、导出完成弹窗和 Rust 单元测试。

本变更在同一个桌面工具中新增“添加水印”工作区。用户需要选择一个图片目录，配置文字或图片水印，然后选择或新建目录导出加水印后的图片。水印图片可能是带颜色和纯色背景的普通图片，需要转换为透明边界后再叠加。

## Goals / Non-Goals

**Goals:**

- 在现有标题区下方增加稳定的 Tab 栏，默认选中 `RAW/JPEG matcher`。
- 将现有 RAW/JPEG 匹配界面与新增水印界面隔离，避免状态互相污染。
- 支持文字水印和图片水印两种来源，并统一进入同一套后端位图叠加流程。
- 对带纯色或近似纯色背景的水印图片提供轻量背景转透明能力。
- 支持批量处理图片目录，并导出到用户选择或新建的目标目录。
- 保持依赖轻量，优先使用 Rust `image` crate 完成图片读写、透明度、模糊和叠加。
- 保持现有 RAW/JPEG 匹配、冲突复核和 RAW 导出行为不变。

**Non-Goals:**

- 不实现 AI 抠图、复杂背景分割或语义级主体识别。
- 不支持 RAW 文件加水印；水印输入目录首版仅处理常见交付图片格式。
- 不实现 EXIF 元数据完整保留承诺；首版以可稳定输出图片为目标。
- 不实现长期水印模板库、云同步或历史任务管理。
- 不改变现有 RAW 导出为扁平目录复制的行为。

## Decisions

### 使用顶层 Tab 状态组织两个工作区

前端新增 `activeWorkspace` 状态，取值为 `raw-jpeg-matcher` 或 `watermark`，默认值为 `raw-jpeg-matcher`。现有三栏匹配界面应抽成 `RawJpegMatcherView`，新增水印界面作为 `WatermarkView`，共同位于 `AppHeader` 下方的 Tab 内容区域。

备选方案是在现有三栏中直接增加水印控件。该方案会让左侧输入、中央结果和右侧统计同时承担两种任务，状态和日志边界不清晰，因此不采用。

### 文字水印先由前端 Canvas 渲染为透明位图

文字水印需要支持中文、系统字体、阴影和透明度。为保持 Rust 依赖轻量，前端使用 WebView Canvas 将文案渲染为透明 PNG/RGBA 小图，并把像素数据随导出配置传给 Rust。Rust 后端不直接负责字体解析，只负责把水印位图叠加到目标图片上。

备选方案是在 Rust 中使用 `ab_glyph` 或 `imageproc` 直接绘制文字。该方案需要处理中文字体文件、字体集合和打包路径，复杂度高于当前需要；如果后续 Canvas 输出质量无法满足，再考虑补充 Rust 字体渲染。

### 图片水印在 Rust 侧做背景转透明

图片水印由用户选择本地图片文件。Rust 后端读取图片后转换为 RGBA，并根据用户选择执行轻量背景去除：

1. 从四角和边缘像素估算背景色。
2. 根据颜色距离和用户配置的容差降低 alpha。
3. 对边缘 alpha 做柔化，减少硬边。
4. 将结果作为统一水印位图继续处理。

该算法适合白底、黑底、纯色底或近似纯色底 logo。复杂照片背景不保证干净去除，应在 UI 中以“自动去背景”而不是“精确抠图”表述。

### 统一水印位图处理管线

无论水印来源是文字 Canvas 位图还是用户图片文件，后端都统一处理为 `WatermarkBitmap`：

```text
文字文案 -> Canvas RGBA/PNG --------┐
                                   ├─ WatermarkBitmap
图片文件 -> 读取 RGBA -> 可选去背景 ┘
          -> 调整尺寸
          -> 调整整体透明度
          -> 生成阴影层
          -> 单点或平铺叠加
          -> 写入导出目录
```

这种设计减少后端分支，便于测试透明度、阴影、文件名冲突和导出摘要。

### 导出目录允许新建且默认不覆盖

水印导出复用现有 RAW 导出习惯：用户点击导出后选择目标目录。目录选择器应启用 `canCreateDirectories: true`，允许在 macOS 对话框中新建目录。后端导出时默认不得覆盖已有文件；目标文件已存在时记录文件名冲突并跳过。

备选方案是自动创建时间戳子目录。该方案能降低冲突，但用户已明确希望“选择目录导出，可新建目录”，因此首版尊重用户选择的最终目录。

### 图片输入保留相对目录结构

水印任务的输入是一个目录，可能包含多层交付图片。导出时建议在目标目录下保留相对于输入根目录的子路径结构，避免不同子目录中的同名图片互相冲突，也便于用户对照原目录。

这与 RAW 导出保持扁平目录不同，因为两个工作流的用户目标不同：RAW 导出是在收集原片，水印导出是在批量生成可交付图片副本。

## Risks / Trade-offs

- 复杂背景图片水印去背景效果不稳定 -> 首版只承诺纯色或近似纯色背景，提供容差和柔化参数，并保留关闭自动去背景的选项。
- Canvas 文字位图传输到 Rust 有序列化成本 -> 水印位图较小，成本可接受；批量处理时只传一次水印位图，不随每张图片重复生成。
- `image` crate 默认格式较多会增加构建体积 -> 使用 `default-features = false` 并显式启用 `jpeg`、`png`，如需 WebP 再单独评估。
- 大目录批量导出耗时较长 -> 后端执行批处理并返回结构化日志；首版可以在导出完成后一次性展示摘要，不要求实时逐文件进度。
- EXIF/色彩配置可能丢失 -> 文档和日志中不承诺完整元数据保留；后续若用户需要再引入专门元数据处理。
- Tab 改造可能影响现有 RAW/JPEG 工作区布局 -> 先抽取视图组件，不改现有业务状态和 Rust 命令行为，并用现有测试验证回归。
