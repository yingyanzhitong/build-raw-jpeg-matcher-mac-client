评分来自启发式审查和运行态验证，并非用户研究或线上遥测。

01 / 执行摘要

先修平台壳层，
再扩展功能。

整体不是“推倒重做”型问题。应用的三工作区组织、流程侧栏和空状态已经形成清楚骨架；优先把窗口、菜单、焦点和外观接入 macOS 约定，就能用相对克制的改动显著提升可信度。

最值得保留的基础

  • 1三工作区信息架构清楚“图片 / RAW 匹配、 一键分离、图片水印”职责分离明确，切换成本低。
  • 2关键状态覆盖较完整空状态、禁用态、进度态和危险确认均有产品化表达。
  • 3基础响应式表现稳定1180×820 与最小 980×680 验证均未出现横向溢出或结构坍塌。

一句话判断

它现在更像一套完成度不错的 Web 工具装进了 macOS 窗口，而不是从 macOS 交互模型出发构建的桌面工具。

首轮整改应集中在平台壳层与无障碍，不建议先做更多装饰性视觉升级。

优势在结构，
短板在平台整合。

04 / 调整后 UI

做减法，
让工具回到内容。

概念稿保持功能、信息结构与中文标签不变，仅重组平台壳层、导航、侧栏和状态区。它用于明确设计方向，不代表代码已经落地。

BEFORE · 当前实现

多层容器 + 自绘平台控件

卡片分隔清楚，但窗口栏、选项卡、统计浮层各自争夺视觉注意力。

AFTER · AI 生成概念

原生层级 + 单一主操作

保留三工作区与五步流程，将统计收回布局流，并降低内容区边框噪声。

01 · 平台壳层标准交通灯与更清晰的标题栏 / 工具栏层级。
02 · 导航表达弱化大胶囊选中态，用紧凑指示器表达当前工作区。
03 · 侧栏结构流程改为来源列表式分组，减少重复卡片边界。
04 · 操作优先级只保留一个系统蓝主按钮，其余操作回归中性层级。
05 · 状态归位统计进入底部状态栏，不再悬浮遮盖列表内容。
06 · 材质克制透明材质仅用于标题栏与导航，内容面保持稳定不透明。
05 / 问题清单

问题可定位，
也可逐项验收。

优先级综合考虑平台违和、可访问性、任务中断概率和改动收益。文件位置是本次审查时的代码证据，后续行号可能随修改漂移。

06 / 整改路线

三段推进，
每段都可验证。

建议先处理平台壳层和阻断型无障碍问题，再统一适配外观，最后优化大规模文件场景。这样能以最小范围的改动先获得最明显的体验提升。

PHASE 01 · P0 / P1

平台与焦点

  • 恢复系统窗口控件或完整模拟 macOS 交通灯行为
  • 为引导弹层与日志抽屉补焦点转移、循环、Escape 和 inert
  • 接入原生应用菜单与核心快捷键
  • 验收：仅键盘完成三个主流程且不会丢失上下文
PHASE 02 · P1

外观与对比度

  • 把固定浅色 token 改为 Light / Dark 自适应语义色
  • 补 Increase Contrast 与 Reduce Transparency 降级
  • 修正状态色和小字号的 4.5:1 对比度
  • 验收：四类系统外观组合均保持可读与层级稳定
PHASE 03 · P2

扩展性与性能

  • 统计区进入正常布局流，消除长列表遮挡
  • 统一 roving tabindex、aria-controls 与方向键模型
  • 对匹配与分离结果列表进行虚拟化 / 分页
  • 验收：千级结果滚动、筛选与预览保持流畅
07 / 方法与边界

证据来自代码，
也来自真实运行。

审查覆盖主界面、三个工作区、关键弹层和窗口缩放；未接入真实 RAW 素材执行完整导出，因此文件错误态、批量任务耗时和真实资源压力不在本次结论范围内。

审查方法

代码静态审查、1180×820 和 980×680 运行态巡检、键盘焦点验证、无障碍语义检查、关键颜色对比度计算，以及基于 Apple Human Interface Guidelines 的平台一致性评估。
