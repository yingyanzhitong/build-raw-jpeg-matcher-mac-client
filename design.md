## Context

本变更新增一个本地 macOS 桌面工具，面向经常收到客户精选 JPEG 文件、并需要收集对应 RAW 原片的摄影师。当前 workspace 中没有该工具的既有应用代码，因此实现时应创建一个新的 Tauri 应用，而不是改造现有产品。

整个工作流完全在本地完成：用户拖入 JPEG 文件或目录，选择 RAW 源目录，执行匹配，处理歧义匹配，然后将选中的 RAW 文件导出到一个目标文件夹。应用应优先保证文件系统行为简单、可预测，而不是引入复杂的元数据匹配。

## Goals / Non-Goals

**Goals:**

- 构建一个基于 Tauri 的 macOS 桌面客户端，并提供实用的上下分区 GUI。
- 支持拖拽输入多个 JPEG 文件和目录。
- 递归扫描 JPEG 输入目录和用户选择的 RAW 源目录。
- 仅使用精确主文件名相等规则匹配 RAW 文件。
- 将歧义 RAW 匹配标记为冲突，并要求用户手动确认。
- 将已匹配和已确认的 RAW 文件导出到用户选择的同一个文件夹。
- 为已找到、未找到、冲突、已确认和已导出的文件提供清晰日志和结果状态。

**Non-Goals:**

- 初始版本不按 EXIF 拍摄时间匹配。
- 初始版本不支持 `-Edit`、`_edited`、`副本` 或修图导出名等模糊后缀规则。
- 不构建图片库管理器、目录数据库或长期运行的索引服务。
- 初始版本不实现内嵌 RAW 解码/预览；冲突复核中可以通过系统默认应用打开 RAW 文件。
- 初始范围不支持 Windows/Linux 打包。

## Decisions

### 使用 Tauri v2、Rust 后端和 TypeScript 前端

前端负责布局、状态、拖拽、目录选择弹窗、结果展示和冲突复核。Rust 命令负责递归文件遍历、扩展名过滤、匹配、复制，以及结构化日志/结果生成。

备选方案：完全在前端通过 Tauri 文件系统 API 实现匹配。该方案会让路径遍历和复制逻辑散落在 UI 代码中，也会让大目录处理更难测试。

### 保持文件名匹配算法简单且确定

初始匹配器应做到：

1. 扩展名比较不区分大小写。
2. 精确比较去扩展名后的主文件名。
3. 根据选定 RAW 根目录构建 RAW 候选映射。
4. 为每个 JPEG 生成四种状态之一：`matched`、`missing`、`conflict` 或 `confirmed`。

如果某个 JPEG 恰好有一个 RAW 候选，则自动匹配；如果没有候选，则标记为未找到；如果存在多个候选，则标记为冲突，并在用户选择一个候选前保持未解决状态。

备选方案：使用修改时间或 EXIF 自动解决冲突。用户已明确偏好在 UI 中手动确认，而不是自动选择。

### 所有选中的 RAW 文件导出到同一个文件夹

导出命令应将每个已匹配或已确认的 RAW 文件复制到同一个用户选择的输出目录。初始版本不保留源目录结构。

如果两个选中的 RAW 文件会产生相同的导出文件名，应用必须避免覆盖，并在日志中报告冲突。安全的初始行为是跳过发生文件名冲突的导出项，并要求用户调整源选择或选择一个干净的导出文件夹。

备选方案：保留源目录结构。用户已明确要求统一导出到一个文件夹。

### 将冲突复核设计为弹窗流程

冲突行应在匹配结果区域以红色视觉状态标记。点击冲突行时打开弹窗，弹窗包含：

- 左侧：相关 JPEG 预览和文件元信息。
- 右侧：RAW 候选列表/网格，包含文件名、路径、扩展名、大小和修改时间。
- 操作：选择候选、打开原始文件、取消。

双击 JPEG 或 RAW 候选时，应通过 Tauri opener 或原生命令使用 macOS 默认应用打开原始文件。

### UI 布局保持紧凑且面向任务

首屏应直接展示可用工具，而不是营销落地页。应用应采用克制的桌面工具布局：

```text
┌─────────────────────────────────────────────────────────────┐
│ JPEG inputs                         RAW source directory     │
│ ┌───────────────────────────────┐  ┌──────────────────────┐ │
│ │ Drop JPG files or folders      │  │ Pick RAW folder       │ │
│ │ Recursive directory scanning   │  │ Recursive RAW scan    │ │
│ └───────────────────────────────┘  └──────────────────────┘ │
├─────────────────────────────────────────────────────────────┤
│ [Find RAW] [Choose Export Folder] [Export RAW Folder] [Clear]│
│ Result summary: matched / missing / conflicts / exported     │
│ Logs and result table                                        │
└─────────────────────────────────────────────────────────────┘
```

## Risks / Trade-offs

- 大型 RAW 目录扫描可能耗时 -> 在 Rust 侧执行匹配，输出进度/日志条目，并保持前端响应。
- 精确同名匹配可能漏掉客户修图后的 JPEG 文件名 -> 初始行为保持简单，并清晰展示未找到结果。
- 多个 RAW 候选可能阻塞导出 -> 使用红色冲突行，并明确要求手动确认。
- 标准 WebView 无法直接显示 RAW 文件 -> 初始版本通过系统默认应用打开 RAW 原片进行检查。
- macOS 文件系统权限可能阻止访问拖入或选择的目录 -> 使用 Tauri dialog/drag-drop 路径和后端命令，并输出清晰权限/错误日志。
- 导出文件名冲突如果处理不当可能覆盖文件 -> 默认永不覆盖，并在日志中报告跳过的冲突项。

## Migration Plan

这是一个新应用，不需要数据迁移。实现时应创建 Tauri 项目，加入匹配/导出工作流，并优先完成本地 macOS 开发运行。

如果实现不被接受，回滚方式是移除新应用目录和对应 OpenSpec 变更。

- 应用是否需要在下次启动时记住上一次使用的 RAW 目录和导出目录？
